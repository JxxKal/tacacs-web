# 0003. SAML SP for admin login with one CLI-managed local break-glass admin

- Date: 2026-05-12
- Status: Accepted

## Context

The web UI itself needs an authentication mechanism for the admin users who configure devices, groups, profiles, and authorizations. Three realistic options:

- **LDAP/AD direct** — same identity store as the TACACS users, simple.
- **SAML SP against the existing FortiAuthenticator** — already operates as the IdP for other internal apps; reuses MFA and conditional-access policies.
- **Local accounts only** — purely managed in our DB.

Whichever path we pick for the day-to-day, the operator can lock themselves out if the IdP is unreachable when they urgently need to reconfigure something (e.g. fix the IdP endpoint after a FortiAuthenticator migration).

## Decision

- Admin login goes through **SAML 2.0 SP integration with FortiAuthenticator as IdP**. SP-Init only, no Single Logout (SLO) in v1.
- Three role tiers — `viewer` / `operator` / `admin` — derived from a configurable mapping of AD-group claim values to roles. Highest matching role wins; no mapping means rejected login.
- Session: HttpOnly + Secure + SameSite=Lax cookie, **8h sliding** expiration with a 24h hard cap.
- **Exactly one** local break-glass admin account, managed only via the CLI subcommand `tacacs-web bootstrap-admin`. Argon2id hash. The UI does not allow creating, listing, or deleting additional local admins.
- Local break-glass login lives at a separate route `/login/local`, prominently styled as emergency-only.
- Every audit-log entry generated by a local-admin session carries `auth_method=local`.
- Optional: pin local-admin login to a configured source CIDR.

## Consequences

- Identity, MFA, and conditional access are owned by FortiAuthenticator; we don't reimplement any of that.
- If FortiAuthenticator is down, the operator can still reach the UI via the break-glass path. Without that, restoring the FortiAuth endpoint config would require direct DB manipulation or a re-deploy.
- We must accept SAML's operational quirks: clock skew, signing-key rotation, IdP metadata drift. These are documented in the setup wizard.
- Refusing UI-managed local accounts removes a tempting parallel user-management system; everyone goes through the IdP except the one emergency hatch.
- The break-glass account has no AD identity and is not subject to the same MFA/conditional access policies. Audit-log marker + optional CIDR restriction are the compensating controls.
- No SLO means logging out of the IdP does not invalidate our session. The 8h sliding cap and the local "Logout" button (which clears our cookie) are deemed sufficient for v1.

## Alternatives considered

- **LDAP/AD direct for admin login** — simpler, but bypasses the centralized MFA + conditional access policies that FortiAuth already enforces for other apps. Would also tightly couple the admin UI to a working AD path, which we explicitly try to avoid in the break-glass design.
- **Multiple UI-managed local admins** — produces two parallel user universes (SAML + local) with their own lifecycle, deletion semantics, and audit trail. Adds complexity without clear benefit; one emergency account is enough.
- **SAML + SLO** — SLO is fragile in practice (each IdP has its own quirks, and back-channel SLO needs additional config). Skipping it limits the surface area; the 8h session cap bounds the risk.
